# slack-keep-active

Небольшой self-hosted сервис, который держит ваш Slack-аккаунт в статусе **active** (🟢 зелёный шарик)
по расписанию рабочих часов. Работает 24/7 в Docker — ноутбук включать не нужно.

## Как это работает

Slack показывает зелёный шарик, пока у вас есть **подключённый активный клиент**. Через официальный
API «зажечь» зелёный нельзя (`users.setPresence` умеет только `auto`/`away`). Поэтому сервис делает то же,
что делает настоящий веб-клиент Slack:

1. Через `rtm.connect` (с токеном веб-сессии `xoxc` + cookie `d`) получает URL WebSocket реального времени.
2. Держит это соединение открытым и шлёт периодические `ping` — Slack видит активного клиента → 🟢.
3. Периодически подтверждает `presence=auto` и (опционально) снимает режим «Не беспокоить», чтобы не было 💤.
4. **Планировщик** включает всё это только в заданные рабочие часы; вне их — отпускает (естественный серый).

### Индикаторы Slack
| Индикатор | Значение |
|---|---|
| 🟢 зелёный | active — есть активный клиент |
| ⚪️ серый | away — клиентов нет или включён ручной away |
| 🟢💤 зелёный с Zzz | active, но включён режим «Не беспокоить» (DND) |

Чтобы был именно сплошной 🟢 без 💤 — поставьте `CLEAR_DND=true`.

## ⚠️ Важные предупреждения

- **Это серая зона.** Имитация присутствия может нарушать правила вашего workspace и/или ToS Slack.
  Используйте на свой риск.
- Авторизация идёт через **токен веб-сессии** (`xoxc-`) и **cookie `d`** (`xoxd-`). Это
  реверс-инжиниринговый, неофициальный способ — Slack может изменить его в любой момент.
- Эти токен и cookie дают **полный доступ к вашему аккаунту**. Храните `.env` как пароль,
  не коммитьте, не передавайте третьим лицам.
- Токен/cookie со временем протухают (особенно после смены пароля или ре-логина) — тогда нужно обновить `.env`.

## Получение токенов

1. Откройте Slack в **браузере** (https://app.slack.com) и залогиньтесь в нужный workspace.
2. Откройте DevTools (F12) → вкладка **Network**.
3. В фильтре наберите `client.boot` (или `api`), обновите страницу (F5).
4. Кликните на запрос → вкладка **Payload** (или **Request**) → найдите параметр `token`
   со значением, начинающимся на `xoxc-`. Это `SLACK_XOXC_TOKEN`.
5. Вкладка **Application** → **Cookies** → `https://app.slack.com` → cookie с именем **`d`**.
   Её значение начинается на `xoxd-`. Это `SLACK_XOXD_COOKIE`.
   - Значение cookie может быть URL-кодировано (`xoxd-...%2F...`) — копируйте как есть.

> Можно автоматизировать извлечение расширениями/скриптами (например, `slack-token-extractor`),
> но ручной способ выше — самый прозрачный.

## Запуск

```bash
cp .env.example .env
# заполните SLACK_XOXC_TOKEN, SLACK_XOXD_COOKIE и расписание в .env

docker compose up -d --build
docker compose logs -f      # проверить, что авторизовались и подключились
```

Остановить: `docker compose down`.

### Без Docker (локально)

```bash
cp .env.example .env   # заполнить
npm install
node --env-file=.env src/index.js
```

## Разработка в команде

Репозиторий рассчитан на нескольких человек, поэтому **секреты в git не попадают**:

1. Клонируйте репозиторий и выполните `npm install` — он же активирует git-хук
   (`core.hooksPath=.githooks`), который блокирует коммит секретов.
2. Скопируйте шаблон: `cp .env.example .env` и заполните **свои** токены. Файл `.env`
   в `.gitignore` — у каждого он свой, в репозиторий не уезжает.
3. Меняете состав переменных? Обновите и `.env.example` (с плейсхолдерами, **без реальных значений**) —
   это единственный источник правды о наборе настроек.

Защита от утечки секретов двухуровневая: `.gitignore` (`.env`, `*.pem`, `*.key`, `.npmrc`, …)
и pre-commit hook `.githooks/pre-commit` — он останавливает коммит при попытке добавить секретный
файл (в т.ч. через `git add -f`) или строку, похожую на токен (`xoxc-`/`xoxd-`, приватные ключи и пр.).
Ложное срабатывание можно обойти осознанно: `git commit --no-verify`.

## Конфигурация (`.env`)

| Переменная | Назначение | По умолчанию |
|---|---|---|
| `SLACK_XOXC_TOKEN` | токен веб-сессии (`xoxc-…`) | — (обязательно) |
| `SLACK_XOXD_COOKIE` | значение cookie `d` (`xoxd-…`) | — (обязательно) |
| `TZ` | таймзона для расписания (IANA) | `Europe/Moscow` |
| `WORK_DAYS` | рабочие дни, `1`=пн…`7`=вс (`1-5` / `1,3,5`) | `1-5` |
| `WORK_START` | начало окна `HH:MM` | `09:00` |
| `WORK_END` | конец окна `HH:MM` (можно через полночь) | `19:00` |
| `JITTER_MIN_MINUTES` | мин. сдвиг плавающих границ, мин | `10` |
| `JITTER_MAX_MINUTES` | макс. сдвиг плавающих границ, мин (`0/0` — выключить) | `15` |
| `CLEAR_DND` | снимать «Не беспокоить» (убрать 💤) | `false` |
| `OFF_HOURS_MODE` | вне часов: `release` или `away` | `release` |
| `PING_INTERVAL_MS` | интервал ping в WebSocket | `20000` |
| `PRESENCE_REFRESH_MS` | как часто подтверждать `presence=auto` | `120000` |
| `HEALTH_PORT` | порт HTTP-эндпоинта `/health` | `3000` |
| `HEALTH_CHECK_MS` | как часто проверять токен через `auth.test` | `300000` |
| `WEBHOOK_URL` | куда POST'ить уведомления об упоминаниях/DM (пусто — только лог) | — |
| `NOTIFY_MENTIONS` | уведомлять о прямых `@`-упоминаниях | `true` |
| `NOTIFY_DM` | уведомлять о любых личных сообщениях (DM) | `true` |
| `NOTIFY_SELF` | уведомлять о собственных сообщениях (для самопроверки) | `false` |
| `LOG_LEVEL` | `debug`/`info`/`warn`/`error` | `info` |

## Плавающее расписание

Чтобы присутствие не включалось/выключалось ровно по часам как робот, границы окна каждый
день случайно сдвигаются на `JITTER_MIN_MINUTES…JITTER_MAX_MINUTES` минут (знак тоже случаен).
Сдвиг **детерминирован по дню**: в течение суток он стабилен (нет мерцания на границе), но
меняется день ото дня. В логах видно сегодняшнее окно: `… (окно сегодня 08:46–18:49)`.

## Мониторинг упоминаний и DM

Пока держится зелёный (т.е. в рабочие часы), сервис слушает тот же WebSocket и реагирует на:

- 🔔 **прямые `@`-упоминания** вас (`<@ваш_id>` в тексте) в любом канале/треде, где вы состоите;
- 🔔 **личные сообщения (DM)** — любое новое сообщение в личке.

Собственные сообщения игнорируются (кроме `NOTIFY_SELF=true` для самопроверки). При срабатывании
имя отправителя резолвится через `users.info`, и формируется уведомление. Если задан `WEBHOOK_URL` —
на него уходит `POST` с JSON, иначе событие пишется только в лог.

> ⚠️ Мониторинг активен **только в рабочие часы** — вне их соединение отпускается (так выбрано в
> конфигурации). Для круглосуточного мониторинга соединение нужно развязать с presence — скажите, доработаю.

### Формат webhook (POST JSON)

```json
{
  "kind": "mention",                 // "mention" или "dm"
  "from": "Vladislav Cherkasov",     // имя отправителя
  "user": "U065FJ9GBFV",             // его user_id
  "channel": "D065J51TS04",          // id канала/DM
  "channel_type": "im",              // "im" (личка) или "channel"
  "text": "<@U065FJ9GBFV> привет",   // текст сообщения
  "ts": "1719267890.123456",         // метка времени Slack
  "permalink": "https://bhftteam.slack.com/archives/D065J51TS04/p1719267890123456"
}
```

Подойдёт любой приёмник: Discord/Telegram-релей, n8n, свой сервер. Для быстрого теста удобно
взять одноразовый URL на https://webhook.site и вписать его в `WEBHOOK_URL`.

## Healthcheck (мониторинг протухания токена)

Сервис периодически (`HEALTH_CHECK_MS`) дёргает `auth.test` и поднимает HTTP-эндпоинт `/health`:

- 🟢 токен валиден → `200 {"status":"ok", ...}`;
- 🔴 токен протух (`invalid_auth`/`token_revoked`/…) → `503 {"status":"token_invalid", ...}`,
  плюс **громкое предупреждение в логах** с инструкцией обновить `.env`.

`Dockerfile` содержит `HEALTHCHECK`, поэтому при протухании контейнер становится **unhealthy**:

```bash
docker inspect --format '{{.State.Health.Status}}' slack-keep-active   # healthy | unhealthy
curl -s http://localhost:3000/health | jq                              # подробности
```

## Если перестало работать

- В логах `auth.test` падает с `invalid_auth`/`not_authed` → токен/cookie протухли, обновите `.env`.
- Шарик не зелёный, хотя сервис подключён → проверьте, что в Slack не выставлен **ручной** «Set yourself
  as away», и что сейчас рабочие часы по `TZ`.
- Виден 💤 → включите `CLEAR_DND=true` (учтите: если у вас в Slack настроено DND-расписание,
  оно будет периодически возвращать DND, сервис будет его снимать).
